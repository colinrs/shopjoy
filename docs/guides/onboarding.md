# Developer Onboarding Guide

Welcome to ShopJoy! This guide will help you set up your development environment and understand the project architecture.

## Prerequisites

- **Go** 1.25+
- **Node.js** 18+ (for frontend development)
- **Docker** & Docker Compose
- **Make** (build automation)
- **MySQL** 8.0+ (or use Docker)
- **Redis** 7+ (or use Docker)

## Quick Start

### 1. Clone the Repository

```bash
git clone https://github.com/colinrs/shopjoy.git
cd shopjoy
```

### 2. Start Infrastructure Services

```bash
# Start MySQL and Redis using Docker
docker-compose up -d mysql redis

# Wait for services to be ready (about 10 seconds)
```

### 3. Initialize Database

```bash
# Connect to MySQL and run init script
mysql -h 127.0.0.1 -P 3306 -u root -p123456 < sql/init.sql
```

### 4. Configure Services

Create configuration files based on templates:

```bash
# Admin API config
cp admin/etc/admin-api.yaml.example admin/etc/admin-api.yaml

# Shop API config
cp shop/etc/shop-api.yaml.example shop/etc/shop-api.yaml
```

Edit the YAML files with your local settings.

### 5. Start Backend Services

```bash
# Terminal 1: Admin API
cd admin && go run admin.go -f etc/admin-api.yaml

# Terminal 2: Shop API
cd shop && go run shop.go -f etc/shop-api.yaml
```

### 6. Start Frontend

```bash
# Terminal 3: Admin Dashboard
cd shop-admin
npm install
npm run dev
```

Access the admin dashboard at: http://localhost:5173

## Project Overview

ShopJoy is a **multi-tenant e-commerce SaaS platform** built with go-zero + DDD (Domain-Driven Design). It features:

- **Admin Dashboard** (`shop-admin/`) — Vue 3 + Element Plus for managing products, orders, users, promotions, and shop settings
- **Customer Storefront** (`joy/`) — Vue 3 + Tailwind CSS for product browsing, cart, checkout, and order tracking
- **Admin API** (`admin/`) — go-zero backend service for admin panel operations
- **Shop API** (`shop/`) — go-zero backend service for customer-facing operations
- **Shared Packages** (`pkg/`) — Common libraries, domain types, authentication, and utilities

**Tech Stack:** Go 1.25, go-zero, GORM, gRPC, etcd, Vue 3, Vite, Pinia, Element Plus, Tailwind CSS, Stripe, Asynq, Docker, GitHub Actions

## Architecture Layers

The codebase is organized into 6 architectural layers:

### 1. Admin API (`admin/`) — 664 files

The primary backend service. Contains HTTP handlers (auto-generated by go-zero), business logic, application services, domain entities, and infrastructure implementations.

Key directories:
- `admin/desc/` — API definition files (`.api` language)
- `admin/internal/handler/` — Auto-generated HTTP handlers (DO NOT edit manually)
- `admin/internal/logic/` — Business logic implementations
- `admin/internal/application/` — Application layer (use case orchestration)
- `admin/internal/domain/` — Domain entities and value objects
- `admin/internal/infrastructure/persistence/` — Repository implementations (GORM)

### 2. Shop API (`shop/`) — 22 files

Customer-facing API service handling cart, orders, and payments. Follows the same DDD structure as Admin API but smaller in scope.

Key domains: `cart`, `order`, `payment`

### 3. Shared Packages (`pkg/`) — 52 files

Cross-cutting concerns used by both Admin and Shop APIs:

| Package | Purpose |
|---------|---------|
| `pkg/auth/` | JWT authentication |
| `pkg/cache/` | Redis caching utilities |
| `pkg/code/` | Standardized error codes (HTTP + business codes) |
| `pkg/contextx/` | Context utilities (tenant, user extraction) |
| `pkg/httpc/` | HTTP client wrapper |
| `pkg/response/` | Unified API response format |
| `pkg/sku/` | SKU code generation and parsing |
| `pkg/snowflake/` | Distributed ID generation |
| `pkg/application/` | Base model with `ID`, `CreatedAt`, `UpdatedAt`, `DeletedAt` |

### 4. Admin Frontend (`shop-admin/`) — 161 files

Vue 3 + Element Plus admin dashboard. Uses Pinia for state management, Vue Router for navigation, and i18n for multi-language support (English/Chinese).

Key areas: `src/views/` (pages), `src/api/` (API clients), `src/components/` (shared components), `src/stores/` (Pinia stores)

### 5. Shop Frontend (`joy/`) — 21 files

Vue 3 + Tailwind CSS customer storefront. Simpler structure focused on product browsing, cart, checkout, and order tracking.

### 6. Database & Schema (`sql/`) — 13 files

SQL schemas organized by domain, each with a `schema.sql` containing all table definitions. ER diagram documented in `sql/er_diagram.md`.

## Guided Tour

Follow these steps in order to build a comprehensive understanding of the codebase:

### Step 1: Project Overview
Start with `README.md` and `docs/ARCHITECTURE.md` to understand the overall design philosophy and tech stack.

### Step 2: API Definitions
Read the `.api` files in `admin/desc/` and `shop/desc/`. These define all routes, request/response types, and middleware using go-zero's API definition language. Start with `admin/desc/admin.api` (the main entry point).

### Step 3: Shared Domain Layer
Explore `pkg/` to understand shared types used across all services:
- `pkg/code/code.go` — All error codes (centralized, with HTTP status + business codes)
- `pkg/application/` — Base model pattern (every entity embeds `application.Model`)
- `pkg/contextx/context.go` — Tenant and user context extraction

### Step 4: Admin Application Layer
Read `admin/internal/application/` to see how use cases are orchestrated. Each subdirectory corresponds to a business domain (user, product, order, fulfillment, payment, promotion, storefront, points).

### Step 5: Admin Domain Entities
Study `admin/internal/domain/` for core business models with their invariants, state machines, and repository interfaces. These define the business rules independent of infrastructure.

### Step 6: Admin HTTP Handlers
Look at `admin/internal/handler/` to see how go-zero auto-generates HTTP handlers from `.api` files. Each handler parses requests and delegates to business logic. **These files are auto-generated — do not edit manually.**

### Step 7: Admin Business Logic
Read `admin/internal/logic/` for the actual business logic implementations. This is where you'll spend most development time when adding features.

### Step 8: Shop API Service
Explore `shop/` to see the customer-facing side: cart management, order creation, and payment processing. Same DDD pattern, smaller scope.

### Step 9: Database Schema
Review `sql/` for table definitions organized by domain. Check `sql/er_diagram.md` for the full entity-relationship diagram.

### Step 10: Admin Frontend
Browse `shop-admin/src/` to see the Vue 3 admin dashboard. Start with `src/router/index.ts` for the route structure, then `src/api/` for API client definitions.

### Step 11: Shop Frontend
Explore `joy/src/` for the customer storefront. Start with `src/router/index.ts` and `src/views/home/index.vue`.

### Step 12: Infrastructure & Deployment
Check `docker-compose.yml`, `Dockerfile`, and `.github/workflows/ci.yml` for containerization and CI/CD setup.

## Key Patterns & Conventions

### DDD Layer Separation

```
Handler (HTTP) → Logic (Business) → Application (Orchestration) → Domain (Entities) → Infrastructure (Persistence)
```

- **Handlers** are auto-generated from `.api` files — never edit manually
- **Logic** files implement business logic for each endpoint
- **Application services** orchestrate complex use cases across multiple domains
- **Domain entities** contain business rules, invariants, and state machines
- **Infrastructure** implements repository interfaces defined in the domain layer

### Multi-Tenancy

Every entity includes `tenant_id` for data isolation. The tenant context is extracted from JWT tokens via `pkg/contextx/`.

### Error Handling

All business errors are defined in `pkg/code/code.go` with:
- HTTP status code (400, 404, 500, etc.)
- Business error code (module-specific ranges like 40xxx for orders)
- Human-readable message

**Never use `errors.New()` in application or domain layers.** Always use `code.ErrXxx`.

### Money & Prices

- API layer: `string` type representing yuan (e.g., `"1.99"` = 1.99 yuan)
- Internal/domain layer: `decimal.Decimal` (from `shopspring/decimal`)
- Database: `DECIMAL(19,4)` columns
- **Never use `float64` for monetary values**

### Time Handling

- Domain entities: `time.Time` type
- Database: `TIMESTAMP` columns (stored as UTC)
- API: RFC3339 strings (e.g., `"2024-03-18T10:00:00Z"`)
- All business time calculations done on backend in UTC

### API Code Generation Workflow

1. Edit `.api` files in `admin/desc/` or `shop/desc/`
2. Run `make api` to regenerate handlers, routes, and types
3. Implement business logic in the generated logic stubs
4. **When backend enums change, update frontend TypeScript types accordingly**

## Development Workflow

### Adding a New Feature

1. **Define the domain entity** in `internal/domain/`
2. **Create repository interface** in the domain package
3. **Implement the repository** in `internal/infrastructure/persistence/`
4. **Create application service** if complex orchestration needed
5. **Define API** in `desc/*.api` and run `make api`
6. **Implement logic handler** in `internal/logic/`
7. **Update frontend** — API types, views, and enum values

### Modifying API Definitions

1. Edit `.api` files in `admin/desc/` or `shop/desc/`
2. Run code generation: `cd admin && make api`
3. Implement the generated logic in `internal/logic/`
4. Update frontend TypeScript types to match

### Code Standards

- **Go**: Follow [Effective Go](https://golang.org/doc/effective_go) and project rules in `.claude/rules/golang/`
- **TypeScript/Vue**: Follow [Vue Style Guide](https://vuejs.org/style-guide/)
- **Commit Messages**: Use conventional commits format
- **Always run `make build`** after code changes to verify compilation
